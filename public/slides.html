Part 1

專案構想

為什麼做 · 解決什麼問題 · 核心設計原則

被丟掉的，其實還能吃

4,000,000 噸

台灣每年浪費的食物

超商每天大量即期品賣不完
宿舍常有一人吃不完的食物
多數沒壞，只是沒遇到對的人

缺的不是食物，而是一個 媒合——把「有多的人」連上「想拿的人」。

這不只是科技問題，更是社會與信任的問題：我們能不能重新讓「和陌生人分享」變得自然？

一張地圖，兩種尺度

超商剩食 — 商業端
即期品打折出清、預約自取
7-ELEVEN + FamilyMart · 全台叢集地圖
宿舍分享 — 個人端
同學之間互相分享剩食
交大 + 清大 共 34 棟宿舍

一邊是店家，一邊是人；同一張地圖、同一套原則。

以地圖為核心降低使用門檻尊重隱私十語國際化

科技之外：這其實是一個「信任」的問題

永續
少丟一份食物，就少一份被白白耗掉的土地、水與人力。
社群
分享食物，是人與人重建信任的方式——讓「開口求助」與「願意幫忙」變得自然。
誠實面對 AI
AI 不會自己解決浪費，它只是降低門檻；真正讓它運作的，是人願意去用、去信任。

接下來想讓大家看：設計怎麼在科技與人之間，架起那道橋。

為什麼選這個架構

網站怎麼做的（前端與部署）

前端

React + TypeScript，用 Vite 建置打包；Tailwind CSS 寫樣式；React-Leaflet 畫互動地圖、標記叢集；React Router 切換兩個模式；自製 Context 管理十語 i18n 與狀態。

後端 API

Cloudflare Pages Functions（檔案即路由），用 TypeScript 寫；資料存在 D1（SQLite），以參數化 SQL 存取，wrangler 管理資料表。

部署 / 架站

一行 wrangler pages deploy 就上線到 Cloudflare Pages；靜態前端與邊緣函式同網域、同平台，不必自建主機或設定伺服器。

工程實務

TypeScript 型別檢查、元件模組化、端側圖片壓縮、AI 輸出防呆過濾；建置全程可重現。

問「我想吃甜的」→ 撈到草莓、蛋糕（字面沒有「甜」字，靠語意）；且只講真的有的，不幻想。

進階：會動手做事的 Agent

對話助手不只回答，還能直接幫你預約（tool-calling）。

「幫我預約誠齋的法國麵包」
→
呼叫 reserve_item 工具
→
原子 claim + 回報聯絡

分兩個模型的原因

  • 一般問答用 8B：快、且不會拒答
  • 預約用 70B：8B 會選錯物品 id
  • 沒預約意圖就不帶工具

四道把關，保護雙方

  • 只有你主動說要預約，AI 才動手
  • 名字得是你親手打的，AI 不能編造
  • 只能預約真實、未過期的物品
  • 同一份只會被領一次，不會被搶走

視覺辨識與即時翻譯

拍照自動辨識
拍一張食物照片，Llama 3.2 Vision 一次讀懂，直接輸出品名與分類。降低上架打字門檻，AI 先填、人確認；失敗自動退回備用模型。
貼文即時翻譯
介面本就十語；連使用者打的貼文也能一鍵翻成觀看者語言（Llama 3.3 70B），呼應大量國際學生的需求。

Part 3

操作與實機展示

leftovermap-nycu.pages.dev

如何使用

① 找剩食
開地圖 → 選超商 / 宿舍模式 → 點標記看清單（品名、效期、預測機率）。
② 上架分享
點宿舍 →「上架物品」→ 拍照按 AI 辨識自動填好 → 確認送出。
③ 領取
點「我要拿這個」→ 填名字 → 看到捐贈人聯絡方式 → 私下約取。
④ 問 AI 助手
右下角按鈕 → 用自然語言問「哪裡有麵包」、或「幫我預約⋯」。

手機掃 QR 即可操作 · 免安裝、免註冊。

實機示範流程

  1. 超商模式：放大新竹 → 點一家店 → 即期品清單 + 供給預測機率（自訓模型）
  2. 宿舍模式：切換 → 點宿舍 → 上架物品 → 拍照 AI 辨識
  3. AI 助手：問「我想吃甜的」→ 以語意撈到草莓、蛋糕
  4. AI 預約：說「幫我預約誠齋的法國麵包」→ 問名字 → 完成預約並回報聯絡
  5. 多語與隱私：切日文整站翻譯 → 聯絡方式只有捐贈人看得到

限制與未來

  • 超商品項目前為模擬資料 → 串接「i 珍食 / 友善食光」官方 API
  • 預測模型以合成資料訓練 → 接真實回報資料重訓，準度才有意義
  • 目前涵蓋交大、清大 → 擴展到更多學校的宿舍
  • 尚無帳號系統 → 加入身分驗證與信任 / 評價 / 檢舉機制
  • RAG 為 brute-force cosine → 規模放大改用 Vectorize 向量索引
